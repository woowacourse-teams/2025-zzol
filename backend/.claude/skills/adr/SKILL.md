`docs/adr/`에 ADR 문서를 작성하고 `docs/adr/index.md`에 한 줄을 추가한다.

## 순서

1. **다음 번호 산정** — `docs/adr/`의 `NNNN-*.md` 파일명 최대값과 `index.md` 행 번호 최대값을 **둘 다** 구해 +1 한다.
   두 최대값이 다르면 파일과 인덱스가 어긋난 것이다. 임의로 진행하지 말고 멈춰서 사용자에게 보고한다 (과거 `0023→0025` 재번호 사고 이력).
2. 주제를 바탕으로 아래 형식에 맞춰 `docs/adr/NNNN-{kebab-case-title}.md`를 작성한다.
3. `index.md` 테이블 맨 끝에 한 줄을 추가한다 (아래 행 형식). 번호 오름차순 유지.
4. **markdownlint 검증** — 저장소 루트에서 `npx markdownlint-cli2@0.22.1`을 실행해 통과시킨다.

## ADR 형식

```markdown
# NNNN. {제목}

- 날짜: YYYY-MM-DD
- 상태: 승인

## 컨텍스트

왜 이 결정이 필요했는가. 해결해야 했던 문제와 당시 상황을 서술한다.

## 결정

무엇을 선택했는가.

## 고려한 대안

| 대안 | 장점 | 단점 |
|------|------|------|
| ...  | ...  | ...  |

## 트레이드오프

선택한 방안의 장단점과 감수한 것들을 서술한다.

## 결과

이 결정이 코드베이스와 팀에 미치는 영향을 서술한다.
```

## 상태(status) 값

작성·갱신 시점에 맞는 값을 쓴다. 이미 구현된 결정을 밋밋하게 `승인`으로만 두지 않는다.

- `승인` — 결정했으나 아직 미구현 또는 구현 중
- `적용됨` — 코드에 반영 완료
- `보류` — 결정했으나 적용 보류
- `폐기` / `대체됨` — 번복. 사유와 대체 ADR 링크를 본문에 추가한다

개정·후속 PR이 있으면 주석을 붙인다: `승인 (2026-04-30 범위 확장)`, `적용됨 (PR #1289 머지)`, `승인 (Phase 1 완료, #1389)`.

## index.md 행 형식

5개 컬럼을 모두 채운다. 번호는 파일 링크로 건다.

```text
| [NNNN](NNNN-title.md) | 제목 | 상태 | 영향 범위 | 핵심 제약 |
```

- **영향 범위**: 이 결정이 적용되는 패키지 또는 도메인 (예: `global/redis`, `cardgame/`)
- **핵심 제약**: 이 결정으로 지켜야 할 규칙을 한 문장으로 (Claude가 충돌 여부를 판단하는 기준이 된다)

> `index.md`는 `.gitattributes`에서 `merge=union`이라 여러 브랜치가 각자 행을 추가해도 자동 병합된다. 맨 끝에 추가만 하면 된다. (union 병합이 순서를 흩뜨리면 리뷰 단계에서만 정렬을 조정한다.)

## 작성 원칙

- 컨텍스트는 "당시 상황"을 기준으로 서술한다. 나중에 읽는 사람이 왜 이 결정을 내렸는지 이해할 수 있어야 한다
- 고려한 대안은 실제로 검토한 것만 작성한다
- 트레이드오프는 단점을 숨기지 않고 솔직하게 작성한다

## Markdown 작성 규칙

Docs CI(`.github/workflows/docs-ci.yml`)가 be/dev PR에서 markdownlint를 **강제**한다. 규칙 SSOT는 저장소 루트의 `.markdownlint.jsonc`이고, 검증도 루트에서 한다(설정 파일이 `backend/`가 아니라 루트에 있다).

ADR은 '고려한 대안' 표와 코드·다이어그램 펜스를 거의 항상 포함하므로, 특히 자주 걸리는 3가지를 점검한다:

- `MD040` — 펜스 코드 블록에 언어 지정 (다이어그램·플로우 등 언어 불명확 시 `text`)
- `MD031` — 펜스 코드 블록 앞뒤에 빈 줄
- `MD022` — 헤딩 앞뒤에 빈 줄

상세 규칙과 예시는 `docs/conventions-docs.md`를 따른다.
