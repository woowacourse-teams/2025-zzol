# design-first

`$ARGUMENTS`에서 기능명을 파싱한다. 이 스킬은 코드를 작성하지 않는다. **모호함 해소 → 설계 → 명세 파일 작성 → 사용자 승인**까지만 수행하고, 구현은 `/impl`에 넘긴다.

## 언제 쓰고 언제 건너뛰는가 (먼저 판단)

- **쓴다**: 새 기능 구현. 결과 diff를 한 문장으로 설명할 수 없는 작업.
- **건너뛴다**: 변경을 한 문장으로 설명할 수 있는 작업(오타·로그 한 줄·변수명·CSS 한 줄 등) → 명세 없이 바로 `/impl`.
- **이 스킬이 아니다**: 버그 수정은 `/fix`(재현 테스트 → 수정). 아키텍처 의사결정 기록은 `/adr`.

> 과설계는 안티패턴이다. 사소한 변경에 명세 루프를 돌리면 순손해다. 애매하면 "한 문장 diff" 규칙으로 판단한다.

## Phase 1: 탐색 (읽기 전용, 사용자에게 묻기 전)

코드베이스에서 스스로 답할 수 있는 것은 사용자에게 묻지 않는다. 먼저 조사한다.

1. `docs/adr/index.md`의 **영향 범위** 컬럼을 기능과 비교한다. 겹치는 ADR이 있으면 읽어 **핵심 제약**을 확인하고, 충돌하면 사용자에게 경고한다.
2. `docs/`에서 기능·도메인 관련 문서(도메인 모델, WebSocket 스펙 등)를 Glob·Grep으로 찾아 읽는다.
3. 관련 도메인의 기존 코드를 Grep으로 탐색해 네이밍·구조 컨벤션과 재사용 가능한 지점을 파악한다.

## Phase 2: 명확화 인터뷰

탐색으로도 해소되지 않은 **진짜 모호한 점만** 사용자에게 묻는다.

- `AskUserQuestion`으로 **한 번에 하나씩** 묻고, **각 질문마다 추천안(기본값)을 함께** 제시한다.
- 확신이 70% 미만인 가정은 추측하지 말고 질문으로 만든다.
- 모호함 유형을 의식적으로 점검한다: 범위(scope), 엣지케이스, 입출력 계약, 실패/에러 동작, 권한, 트레이드오프.
- 인터뷰로도 끝내 확정 못 한 항목은 명세에 `[NEEDS CLARIFICATION: 구체적 질문]` 마커로 남긴다. 추측으로 메우지 않는다.

## Phase 3: 명세 작성

`docs/specs/{kebab-case-기능명}.md`에 아래 형식으로 작성한다. 계층 구조는 `/impl`의 도메인 → 서비스 → 컨트롤러 흐름에 맞춘다.

```markdown
# {기능명} 설계

- 날짜: YYYY-MM-DD
- 상태: 초안
- 관련 ADR: {있으면 NNNN, 없으면 없음}

## 목표

이 기능이 무엇을 해결하는가. 한두 문장.

## 명확화 결과

- Q: {물었던 질문} → A: {확정된 답}
- ...

## 범위

- 포함: ...
- 제외(이번에 안 함): ...

## 설계

### 도메인

엔티티/VO/도메인 이벤트/불변식. 순수 Java(Spring·JPA 의존 없음).

### 서비스

유스케이스 조합, 트랜잭션 경계, 이벤트 발행 위치.

### 컨트롤러 / 메시지

REST 엔드포인트 또는 WebSocket/Redis Stream 경로. (메시지 흐름은 Stream 경유 필수)

## 성공 기준 (checkable)

- [ ] {해석 여지 없이 검증 가능한 기준. 예: `NicknameTest 통과`, `PUT /nickname → 200`}
- [ ] ...

## 미해결 질문

- [NEEDS CLARIFICATION: ...] (없으면 "없음")
```

작성 원칙:

- **성공 기준은 checkable하게.** "잘 동작한다" 대신 "테스트 X 통과", "엔드포인트가 200 반환"처럼 해석 여지 없이 검증 가능한 문장으로 쓴다.
- 요청된 기능만 설계한다. "나중에 필요할 것 같은" 추상화는 명세에 넣지 않는다.
- 아키텍처 핵심 제약(모듈 의존 방향, Stream 경유)을 위반하지 않는지 설계 단계에서 확인한다.

## Phase 4: 승인 & 인계

1. 작성한 명세 파일 경로와 핵심 결정(명확화 결과·성공 기준)을 사용자에게 요약한다.
2. **사용자 승인을 받는다.** 미해결 질문(`[NEEDS CLARIFICATION]`)이 남아 있으면 구현으로 넘어가지 말고 먼저 해소한다.
3. 승인되면 상태를 `초안` → `승인`으로 바꾸고, 다음을 안내한다: `/impl {기능명}` 으로 이 명세를 읽어 구현.

## Markdown 작성 규칙

명세 문서는 markdownlint 검사를 통과해야 한다. 규칙은 `docs/conventions-docs.md`를 따른다.
