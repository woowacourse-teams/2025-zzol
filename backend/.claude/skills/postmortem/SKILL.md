`docs/postmortem/` 디렉토리에 포스트모템 문서를 작성한다.

ADR과 포스트모템은 장르가 다르다. **ADR은 "앞으로 무엇을 할지"의 결정 기록(미래지향·불변)**이고, **포스트모템은 "무엇이 어떻게 터졌는지"의 복기 기록(과거지향·회고)**이다. 회고에서 새 결정이 도출되면 그 결정은 `/adr`로 별도 작성하고 양쪽을 상호 링크한다. 진단이 빗나갔거나 결정이 번복된 과정 자체가 포스트모템의 핵심 자산이므로 숨기지 않는다.

## 순서

1. **다음 번호 산정** — `docs/postmortem/`의 `NNNN-*.md` 파일명 최대값과 `index.md` 행 번호 최대값을 **둘 다** 구해 +1 한다 (번호는 ADR과 독립된 시퀀스, 비어 있으면 0001부터). 두 최대값이 다르면 파일과 인덱스가 어긋난 것이니 멈추고 사용자에게 보고한다.
2. 사건을 바탕으로 아래 형식에 맞춰 포스트모템을 작성한다
3. 파일명은 `docs/postmortem/NNNN-{kebab-case-title}.md` 형식으로 생성한다
4. 사실관계는 메모리·추측이 아니라 git 로그·PR·실제 코드로 검증한 뒤 서술한다. 확인 못 한 부분은 "미해결/미확인"으로 명시한다
5. 회고에서 결정이 도출됐으면 `/adr`로 ADR을 작성하고, 포스트모템 상단 메타의 `관련 ADR`에 링크한다
6. `index.md` 테이블 맨 끝에 한 줄을 추가한다 (아래 `index.md 행 형식`). 번호 오름차순 유지.
7. **markdownlint 검증** — 저장소 루트에서 `npx markdownlint-cli2@0.22.1`을 실행해 통과시킨다.

## 포스트모템 형식

```markdown
# NNNN. {제목}

- 날짜: YYYY-MM-DD
- 심각도: P0 | P1 | P2
- 상태: 작성중 | 완료 | 미결
- 관련 ADR: [NNNN 제목](../adr/NNNN-....md)  (회고에서 결정이 나왔으면 링크, 없으면 줄 생략)
- 관련 포스트모템: [NNNN 제목](NNNN-....md)  (연결된 회고가 있으면, 없으면 줄 생략)

## 요약

무엇이 언제 터졌고, 임팩트와 결론은 무엇인가. 한 문단.

| 시점 | 사건 |

| ...  | ...  |

## 임팩트

영향 범위 — 사용자·CI·배포·데이터에 미친 실제 피해.

## 근본 원인

표면 증상이 아니라 메커니즘. 왜 일어났는가. 진단이 빗나갔다면 어떤 가설을 어떤 근거로 폐기했는지 함께 적는다.
```

## index.md 행 형식

6개 컬럼을 모두 채운다. 번호는 파일 링크로 건다.

```text
| [NNNN](NNNN-title.md) | 제목 | YYYY-MM-DD | P0|P1|P2 | 근본 원인 한 문장 | [NNNN](../adr/NNNN-....md) |
```

- **심각도**: P0(사용자 영향 프로덕션 장애) / P1(배포·운영 중단·데이터 위험) / P2(개발·CI 생산성 저하)
- **근본 원인**: 표면 증상이 아니라 메커니즘을 한 문장으로
- **관련 ADR**: 회고에서 결정이 나왔으면 링크, 없으면 비운다

> `index.md`는 `.gitattributes`에서 `merge=union`이라 여러 브랜치가 각자 행을 추가해도 자동 병합된다. 맨 끝에 추가만 하면 된다.

## 작성 원칙

- **비난하지 않는다(blameless).** 사람이 아니라 시스템·프로세스·정보 부족을 분석한다. "누가 틀렸나"가 아니라 "왜 그 시점에 그 판단이 합리적으로 보였나"를 적는다
- **오진을 숨기지 않는다.** 빗나간 가설과 폐기 근거가 포스트모템의 핵심 가치다. "처음엔 X를 의심했으나 Y로 폐기"를 명시한다
- **근본 원인은 표면 증상과 구분한다.** 로그 에러 메시지가 아니라 그 메시지를 만든 메커니즘까지 내려간다
- **결론의 강도를 근거의 강도에 맞춘다.** 재현·검증이 없으면 "확정"이 아니라 "추정/확인 예정"으로 적는다
- **타임라인은 시점을 검증 가능한 단위(커밋·PR·시각)로 적는다.** 정확한 시각을 모르면 지어내지 말고 PR/커밋 단위로 적는다
- **액션 아이템은 검증 가능하게.** "조심한다"가 아니라 "재현 매트릭스를 확보한 뒤 복구"처럼 완료 판정이 가능하게 쓴다

## Markdown 작성 규칙

Docs CI(`.github/workflows/docs-ci.yml`)가 be/dev PR에서 markdownlint를 **강제**한다. 규칙 SSOT는 저장소 루트의 `.markdownlint.jsonc`이고, 검증도 루트에서 한다(설정 파일이 `backend/`가 아니라 루트에 있다).

포스트모템은 타임라인·액션 등 **표가 여러 개**이고 로그·코드 펜스도 흔하므로, 특히 자주 걸리는 3가지를 점검한다:

- `MD040` — 펜스 코드 블록에 언어 지정 (로그·플로우 등 언어 불명확 시 `text`)
- `MD031` — 펜스 코드 블록 앞뒤에 빈 줄
- `MD022` — 헤딩 앞뒤에 빈 줄

상세 규칙과 예시는 `docs/conventions-docs.md`를 따른다.
