# 0001. AI 닉네임 검열 + 운영자 피드백 루프 설계

- 날짜: 2026-03-21
- 상태: 승인 (2026-04-30 범위 확장)

## 컨텍스트

입장 시 비속어 필터(`bad-word-filtering`)는 직접적인 욕설은 잡지만 특수문자 삽입(`씨*발`), 유사 발음 대체(`씌발`), 초성 우회(`ㅅㅂ`) 같은 창의적 우회는 탐지하지 못한다. 랭킹에 노출되는 닉네임은 불특정 다수에게 보이기 때문에, 방에 입장할 수 있었던 닉네임이라도 랭킹 노출 전에 별도로 검열할 필요가 있었다.

AI 판단은 오판 가능성이 있으므로 신뢰도(confidence)에 따라 처리를 분리하는 구조를 선택했다. 신뢰도가 높은 경우(≥ 0.85)는 자동 차단 후 어드민이 이의 제기 시 해제하고, 신뢰도가 낮은 경우(< 0.85)는 운영자가 직접 판단하게 한다. 또한 운영자가 AI 오판을 정정한 사례를 모아 다음 검열에 활용하는 피드백 루프로 시간이 지날수록 정확도를 높인다.

## 결정

- **AI 모델**: Gemini 2.5 Flash (무료 티어, RPM 5)
- **검열 시점**: 룰렛 결과 저장 시 닉네임을 등록, 12시간 주기 스케줄러가 배치 처리
- **결과 분류 및 처리**:

| 상태        | 조건                                   | 자동 처리                                                  | 운영자 액션                |
|-----------|--------------------------------------|--------------------------------------------------------|-----------------------|
| `FLAGGED` | `flagged=true` & `confidence ≥ 0.85` | `custom_profanity` 등록 + `BadWordFiltering.add()` 즉시 차단 | 허용(ALLOWED) 시 필터에서 제거 |
| `PENDING` | `flagged=true` & `confidence < 0.85` | 없음 (대시보드 보류 목록 노출)                                     | 허용 또는 차단 직접 결정        |
| `CLEAN`   | `flagged=false`                      | 없음                                                     | —                     |

- **피드백 주입**: 운영자 결정이 누적 20건 이상이면 다음 Gemini 프롬프트에 few-shot 예시로 동적 삽입
- **파싱 실패 전략**: JSON 전체 파싱 실패 시 해당 배치 skip(다음 실행에서 재시도), 항목 단위 파싱 실패 시 해당 항목만 skip

## 고려한 대안

| 대안                         | 장점               | 단점                                        |
|----------------------------|------------------|-------------------------------------------|
| confidence 무관하게 전량 운영자 검토  | 오판으로 인한 차단 없음    | 운영자 부담 과다, 대시보드 방치 시 문제 닉네임이 랭킹에 노출된 채 유지 |
| confidence 무관하게 전량 자동 차단   | 운영자 개입 불필요       | AI 오판 시 정상 닉네임 차단, 복구 절차 필요               |
| 실시간 검열 (입장 시 즉시 Gemini 호출) | 즉각 반영            | 무료 티어 RPM 5 한계, 입장 응답 지연 유발               |
| 자체 분류 모델 학습                | 도메인 특화, 추론 비용 없음 | 학습 데이터·인프라·유지보수 비용 과다                     |
| 운영자 전담 수동 검열               | AI 오판 없음         | 확장성 없음, 운영 부담                             |

## 트레이드오프

**감수한 것들**

- 검열이 사후 처리이므로 룰렛 결과 저장 시점과 실제 차단 시점 사이에 최대 12시간의 갭이 존재한다. 그 사이 랭킹에 문제 닉네임이 노출될 수 있다.
- FLAGGED 자동 차단은 AI 오판 시 정상 닉네임이 운영자 해제 전까지 실제로 차단된다. confidence 0.85 기준으로 오판율을 낮추지만 0은 아니다.
- few-shot 피드백 주입은 20건 미만 구간에서는 동작하지 않아 초기 정확도 개선 효과가 없다.
- Gemini 무료 티어 RPM 5 제한으로 대량 적체 시 처리 속도가 느리다 (100건/배치 × 4.6회/분 = 약 460건/분).

**얻은 것들**

- 신뢰도 기준으로 자동 차단 / 수동 검토를 분리해 운영자 개입 비용을 줄이면서도 오판 리스크를 관리한다.
- FLAGGED 자동 차단이 입장 시 비속어 필터(`BadWordFiltering`)에도 즉시 반영되어 우회 패턴이 재사용될 때 실시간으로 차단한다.
- 운영자 결정이 쌓일수록 프롬프트 품질이 올라가는 자기 개선 구조다.

## 결과

- `room/infra/persistence/nickname/` 하위에 3개 테이블(`nickname_audit`, `nickname_feedback`, `custom_profanity`) 추가
- `GeminiPlayerNameAuditor`는 `!local & !test` 프로파일에서만 활성화. `local`·`test` 환경에서는 `NoOpPlayerNameAuditor`(항상 CLEAN 반환)가 대신 등록된다
- 운영자 대시보드(`/admin/playername-audit`)는 Spring Security + Thymeleaf로 구현. `/admin/**` 외 기존 REST API·WebSocket 엔드포인트는 모두 `permitAll()`로 열려 있다
- `NicknameAuditProperties`에서 `flagged-threshold`(기본 0.85)와 `feedback-injection-threshold`(기본 20)를 조정할 수 있다
- Resilience4j RateLimiter(`limit-refresh-period: 13s`)로 Gemini RPM 5 한계를 선언적으로 제어한다

## 범위 확장 (2026-04-30)

ADR 0003(OAuth2 로그인 도입)으로 회원 가입·프로필 닉네임 변경 시점이 검열 트리거에 추가되었다.

- **회원 가입 시**: OAuth provider 닉네임을 기본값으로 사용하되, `PlayerNameValidator` 즉시 검사 통과 후 `PlayerNameAuditService`에 AI 사후 검열 등록.
- **닉네임 변경 시** (`PATCH /users/me/nickname`): 동일하게 즉시 검사 + AI 사후 검열.
- 기존 "룰렛 결과 저장 시점" 트리거는 그대로 유지된다.
