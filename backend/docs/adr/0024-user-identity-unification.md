# 0024. 사용자 식별 통합 — 중복 닉네임 허용과 `name#UserCode` 전역 식별자

- 날짜: 2026-06-09
- 상태: 제안 (설계 논의 단계, 미결 질문 존재)
- 관련: ADR-0025(Room-GameSession 분리 — 이 결정을 표면화한 트리거), ADR-0006(UserCode 정책), ADR-0009(WebSocket Room Session Token / PlayerKey)

## 컨텍스트

현재 방은 입장 시 닉네임 중복을 차단한다(`Room.validatePlayerNameNotDuplicate` → `DUPLICATE_PLAYER_NAME`, 409). 이 덕분에 방 안에서 이름이 곧 유니크 식별자이고, 호스트 검증을 비롯한 모든 플레이어 식별이 이름 기준(`sameName`)으로 동작한다. ADR-0025의 `GameSession` 호스트 검증도 이 불변식에 기대 이름 기반으로 설계됐다.

그러나 이 닉네임 유니크 강제는 **로직 복잡도를 피하려는 임시 조치**일 뿐 영구 보장이 아니다. 다음 두 변경이 예정돼 있고, 서로 맞물려 있어 한 묶음으로 결정해야 한다.

1. **중복 닉네임 허용** — 차후 리팩터로 방 안에서도 같은 닉네임을 허용할 예정이다. 허용되면 이름은 더 이상 방-유니크 식별자가 아니므로, 이름에 기대던 모든 식별 경로(ADR-0025의 `GameSession` 포함)가 깨진다.
2. **전역 식별자 `UserCode`** — 프론트엔드는 전역 범위에서 닉네임이 중복되므로 `name#UserCode`(ADR-0006, 5자리 불변)로 사용자를 식별하기로 확정했다. 내부 PK인 `userId`는 외부에 노출하지 않는다.

핵심 난점은 **게스트**다. `UserCode`는 회원 가입 시 발급되므로(ADR-0006) 게스트는 `UserCode`가 없다. 따라서 중복 닉네임이 허용되는 순간 게스트를 유일하게 가리킬 키가 사라진다.

## 결정

확정된 방향은 다음과 같다. 세부 미결 사항은 아래 "미결 질문"에서 다룬다.

### 1. 전역 식별 표현은 `name#UserCode`

프론트엔드는 사용자를 `name#UserCode`로 식별·표시한다. 닉네임이 충돌해도 `UserCode`가 회원을 유일하게 구분한다.

### 2. `userId`는 외부에 노출하지 않는다

`userId`는 내부 DB PK다. `Gamer`가 `:game-api` 이벤트·WebSocket 페이로드로 흐르는 지점을 점검해 `userId` 누출을 막는다. 외부로 나가는 신원 표현은 `UserCode`(회원) 또는 게스트 키로 통일한다.

### 3. 식별자와 타입을 한 값객체로 묶는다 — `GamerId`

"회원이냐 게스트냐(타입)"와 "어느 플레이어냐(식별자)"는 별개 개념이다. 이를 분리하되 한 값객체로 묶어, `userCode == null` 같은 암묵적 분기를 제거하고 모든 플레이어가 non-null 키를 갖게 한다.

```text
GamerId {
    type: MEMBER | GUEST,        // 타입 — 메타데이터/렌더링용
    key:  String (non-null)      // 식별자 — MEMBER=UserCode, GUEST=게스트 키(미결)
}
// equals / Map 키 = (type, key). null 키 없음.
```

- `type`은 프론트가 회원/게스트를 구분해 렌더링하는 데 쓰며, 식별자가 아니다.
- `key`는 방 안에서 유니크하고 세션 동안 안정적이어야 한다. 회원은 `UserCode`, 게스트는 아래 미결 질문에서 정한다.

## 미결 질문

### Q1. 게스트 키를 무엇으로 둘 것인가

| 대안 | 게스트 키 | 중복 게스트 이름 | 충돌 | 비고 |
|------|-----------|------------------|------|------|
| A. `PlayerKey` 재사용 | `PlayerKey`(ADR-0009) | 허용 | 결정적 0 | 방-유니크 토큰이 이미 존재. 새 필드 불필요 |
| B. 입장 시 랜덤 토큰 | 랜덤 값 | 허용 | 확률적(작음) | A의 재발명. 결정적 유니크를 확률적으로 후퇴 |
| C. CODE=`GUEST` 스톱갭 | 이름 | **불가** | 결정적 0 | 게스트만 종전 유니크 유지. 노력 최소, 절반 완화 |

- **A 권고**: 게스트마다 방-유니크 토큰을 부여하는 일은 `PlayerKey`가 이미 한다. ≤9명 방에서 결정적 키는 충돌이 정확히 0이라 확률 계산이 필요 없다. B(랜덤)는 추측 불가능성이 필요할 때만 정당한데 방 안 식별엔 불필요하므로 채택하지 않는다.
- **C는 단기 스톱갭**: "한 방에 같은 닉네임 게스트 둘이 동시에 들어와야 하는가"가 아니오라면, 회원만 `UserCode`로 풀고 게스트는 종전 이름 유니크를 유지하는 최소 변경이 가능하다.
- 결정 기준: **한 방에 같은 닉네임 게스트가 동시에 들어와야 하는가?** 예 → A, 아니오(또는 추후) → C.

### Q2. 게스트 재접속 연속성

게스트가 끊고 재입장할 때 같은 키를 유지해야 점수·식별이 이어진다. 키 선택이 이 동작을 좌우한다.

- A(`PlayerKey`): 토큰이 재발급되면 식별·점수가 끊긴다. 재접속 시 토큰 복원 또는 매핑 전략이 필요하다.
- C(이름): 같은 이름으로 다시 들어오면 동일 식별이라 재접속에 유리하다.

### Q3. `PlayerKey`의 모듈 위치 (`:room` 결합)

`PlayerKey`는 `:room` 타입이다. 이를 게임 내부 식별 키로 쓰면 ADR-0025이 제거한 `:game → :room` 의존이 되살아난다. A를 택할 경우 `PlayerKey`(또는 그 값)를 `:game-api`/`:common`으로 올리거나 래핑해야 한다.

## 고려한 대안 (전역 식별 표현)

| 대안 | 장점 | 단점 |
|------|------|------|
| `name#UserCode` (채택) | 전역 유니크, 불변, 외부 노출에 안전 | 게스트는 `UserCode` 부재로 별도 키 필요 |
| `userId` 직접 노출 | 변환 비용 없음, 항상 존재 | 내부 PK 외부 노출, UserCode 전환 시 이중 마이그레이션 |
| 이름만 사용 | 단순 | 전역에서 닉네임 충돌, 식별 불가 |

## 트레이드오프

- `GamerId` 도입은 모든 `Gamer` 생성 지점과 WebSocket 컨트랙트, 친구·프로필 도메인의 식별 표현을 함께 바꿔야 하므로 변경 범위가 넓다. 대신 식별 의미가 명시화되고(`null` 키 제거) 내부 PK 노출이 막힌다.
- 게스트 키를 A(`PlayerKey`)로 통일하면 회원/게스트를 단일 추상으로 다루지만 `:room` 결합 해소와 재접속 연속성 설계 비용이 따른다. C(스톱갭)는 비용이 작지만 게스트 중복 닉네임을 포기하는 절반짜리 완화다.

## 결과

- ADR-0025의 `GameSession` 이름 기반 호스트/플레이어 식별은 본 결정의 식별 모델(`GamerId`)로 이관된다. ADR-0025은 그 전까지 유효한 이름 기반 식별로 진행하며, 본 ADR이 적용되면 해당 ⚠️ 표시 지점이 해소된다.
- 중복 닉네임 차단(`DUPLICATE_PLAYER_NAME`)은 회원에 한해 해제되며, 게스트는 Q1 결정에 따라 유지(C) 또는 해제(A)된다.
- 이 ADR은 미결 질문(Q1~Q3)이 닫히면 상태를 `승인`으로 전환한다. Q1은 프론트·기획의 "게스트 중복 닉네임 허용 여부" 확정이 선행돼야 한다.
