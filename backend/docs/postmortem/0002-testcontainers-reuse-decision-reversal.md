# 0002. Testcontainers reuse 의사결정 2회 번복

- 날짜: 2026-06-18
- 심각도: P2 (CI 생산성·의사결정 비용, 사용자 영향 없음)
- 상태: 미결 (reuse 복구 보류)
- 관련 ADR: [0013 도메인 모듈 테스트 독립 실행 전략](../adr/0013-domain-module-test-isolation.md)
- 관련 포스트모템: [0001 게임 통합테스트 플레이키](0001-game-integration-test-flaky.md)

## 요약

[포스트모템 0001](0001-game-integration-test-flaky.md)의 게임 통합테스트 플레이키를 두고, 근본 원인을 "공유 컨테이너 reuse"로 귀속해 **reuse를 영구 비활성화하고 모듈별 DB/Redis 격리까지 제거**하는 결정을 내렸다(#1402/#1403). 이후 reuse 자체는 원인이 아니며 유일한 새 변수는 타이밍 결함 수정(#1411)이었음이 드러나 결정을 철회하고 reuse를 복구했다(#1418). 그러나 복구 PR마저 검증 부족으로 revert되어(`1e9cde36`), 현재 `be/dev`는 다시 reuse OFF 상태다. 한 인시던트의 조치 결정이 **두 번 뒤집힌** 프로세스 회고다.

## 임팩트

- 동일 토글(`withReuse`)을 제거 → 복구 → 재제거로 세 차례 왕복. ADR-0013 문서도 같은 횟수만큼 개정·되돌림.
- CI는 reuse OFF로 안정적이나 모듈별 컨테이너 중복 기동으로 **빌드 시간 최적화(#1364)를 일부 반납**하는 트레이드오프가 미해소 상태로 남음.
- 사용자 영향 없음.

## 근본 원인

프로세스 차원의 실패가 겹쳤다.

- **근본 원인 오귀속.** 플레이키의 메커니즘(stale 이벤트 → 컨슈머 포화)에서 reuse는 _필요조건_이었을 뿐 _충분한 단독 원인_이 아니었다. "reuse를 끄면 통과한다"는 관찰을 "reuse가 원인이다"로 비약해 격리 인프라까지 제거했다.
- **변경 변수 통제 실패.** #1403 비활성 시점(06-11)에 이미 #1361(스트림 레지스트리)·#1369(컨텍스트 캐시 정리)가 트리에 있었고 그 상태에서도 reuse-on 플레이키가 재현됐다. 따라서 #1361을 "이제 안전" 근거로 삼으면 기록과 모순된다. 실제로 새로 들어온 변수는 #1411 단 하나였는데, 여러 변경이 동시에 트리에 섞여 무엇이 진짜 수정인지 분리되지 않았다.
- **재현 환경 제약.** 로컬 Docker 미가용(win32)으로 복구의 안전성을 로컬에서 매트릭스로 검증할 수 없었고, "CI로 확인" 프레이밍 대신 "safe" 단정으로 넘어간 것이 #1418 revert로 이어졌다.

## 재발 방지 액션

| 액션 | 상태 |
|------|------|
| 플레이키 진단 시 변경을 **한 번에 하나씩** 격리해 실제 수정 변수를 분리 | ☐ 프로세스 합의 필요 |
| reuse 복구는 CI에서 이전 플레이키 게임 IT(RacingGame/BlindTimer/BlockStacking/CardGame/Ladder) 재발 매트릭스를 확보한 뒤 진행 | ☐ 미결 |
| "영구 결정"·"safe" 같은 단정 표현은 재현/검증 근거가 있을 때만 사용 | ☐ 프로세스 합의 필요 |
| ADR-0013의 reuse 정책 서술을 현재 상태(OFF, 복구 보류)와 일치시킴 | ☑ index 요약 정정 + 본 회고 역링크 추가 |

## 교훈

- **"끄면 통과한다"는 원인 증명이 아니다.** 비활성화가 증상을 가린 것인지 원인을 제거한 것인지 구분하려면, 비활성화한 변수 외 다른 변수를 고정한 재현이 필요하다.
- **결정을 내리기 전에 "이 결론이 기존 기록과 모순되지 않는가"를 점검한다.** #1361이 이미 트리에 있었다는 사실 하나가 "reuse가 단독 원인"이라는 결론을 무너뜨렸다.
- **번복 가능한 결정에 "영구"라는 라벨을 붙이지 않는다.** 라벨이 강할수록 되돌리는 비용(코드 + 문서 + 합의)이 커진다.
- 환경 제약(로컬 재현 불가)을 인지했다면 **결론의 강도를 그 제약에 맞춰 낮춘다** — "확정"이 아니라 "CI로 확인 예정".
