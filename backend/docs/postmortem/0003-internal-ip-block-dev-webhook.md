# 0003. 내부 IP 차단 인시던트 — dev webhook 테스트가 prod 리스너로 유입 + 화이트리스트 유실

- 날짜: 2026-06-23
- 심각도: P1
- 상태: 작성중
- 관련 ADR: [0032 zzol-bot을 Alertmanager 웹훅 수신기로 재배치](../adr/0032-zzolbot-as-alertmanager-receiver.md)

## 요약

dev 환경의 zzol-bot 웹훅 수신기를 수동 테스트하려 했으나, nginx 내부 전용 리스너가 prod 전용(`8889` → `prod-service.inc` → prod-app)으로만 설정돼 있어 테스트 요청이 prod-app으로 유입됐다. prod-app이 404를 반환했고, `IpBlockFilter`의 404 누적 차단 로직이 요청 출발지인 nginx의 내부 도커 IP(`172.21.0.5`)를 임계값(5회) 초과로 차단했다. 이 차단을 막았어야 할 내부 IP 화이트리스트(#1352)는 구현됐으나 어느 보호 브랜치에도 머지되지 않은 상태여서 가드가 부재했다. 결과적으로 nginx를 경유하는 내부 트래픽이 403으로 거절됐고, `prod-redis`의 `block:*` 키를 수동 삭제해 완화했다. **단일 결함이 아니라 두 결함의 합작**이다: (1) 내부 리스너의 dev/prod 환경 미분리, (2) 머지되지 않아 유실된 화이트리스트.

## 임팩트

- 영향 대상: `prod-app`이 차단된 출발지 IP로 인식한 nginx 경유 트래픽. nginx가 prod-app 앞단 프록시이므로, 차단이 지속됐다면 nginx를 통한 prod HTTP 요청 전반이 403으로 거절될 수 있는 구조였다.
- 실제 사용자 피해 범위(외부 사용자 트래픽까지 도달했는지)는 **미확인** — 수동 테스트 중 조기에 인지·완화했다. 차단 대상이 사용자 대면 활성 색의 egress IP였다면 P0로 격상될 수 있었다.
- 데이터 손실 없음. `block:*` 키만 삭제했고 룸/세션/스트림 등 다른 Redis 키는 보존했다(`FLUSHALL` 미사용).

## 근본 원인

표면 증상은 "내부 IP가 차단됨"이지만, 두 개의 독립 결함이 맞물려 발생했다.

1. **내부 리스너의 환경 미분리.** `internal.conf`는 `8889` 단일 리스너에서 `prod-service.inc`(`$upstream` = prod 활성 색)만 include한다. dev 웹훅을 테스트할 경로가 없어, 테스트 요청이 prod-app으로 유입됐다. prod-app이 404를 반환한 정확한 원인(경로/페이로드/토큰 불일치)은 **미확인**이나, 404가 반환됐다는 사실과 그 결과는 확인됐다.

2. **머지되지 않아 유실된 화이트리스트(핵심 증폭 원인).** `IpBlockFilter`의 404 누적 차단(`incrementNotFoundAndBlockIfExceeded`)은 출발지 IP를 `getRemoteAddr()`로 얻는다. 프록시 뒤에서 이 값은 nginx의 내부 도커 IP다. #1352가 추가한 `isInternalIp` 가드가 있었다면 내부 IP는 누적·차단 대상에서 제외됐겠지만, 그 코드는 `be/dev`·`be/prod`·`main` 어디에도 머지되지 않았다(2026-06-23 `git merge-base` 확인). 같은 파일을 동시에 건드린 #1353(Ip 값 객체 리팩터)만 머지되면서, 두 PR의 의도가 합쳐지지 못하고 화이트리스트가 조용히 사라졌다.

진단 과정에서 처음에는 "필터 로직 자체의 버그"를 의심했으나, git 이력 추적 결과 **로직은 올바르게 작성됐고 머지 단계에서 유실**된 것으로 확인해 가설을 교정했다.

## 대응

- 탐지: dev 웹훅 수동 테스트 중 내부 트래픽 403 발생으로 인지.
- 완화: `prod-redis`에서 `block:ip:*`·`block:404:*` 키(`block:*`)를 SCAN 기반으로 삭제. 어드민 해제 API도 시도했으나 호출 출처(`api.zzol.site`)가 CORS 허용 목록에 없어 거절됨.
- 근본 수정: 이슈 #1493 / PR #1494로 처리(nginx 8890 dev 리스너 분리 + 내부 IP 화이트리스트 복구 + 어드민 CORS). dev 웹훅 경로는 8890 e2e로 검증 완료 — 절차·함정은 [ADR-0032](../adr/0032-zzolbot-as-alertmanager-receiver.md)의 "배포 후 검증 › dev 수동 e2e 검증 절차" 참조.

| 같은 파일을 건드리는 병렬 PR이 서로의 변경을 덮어쓰지 않도록, 머지 전 대상 파일의 양쪽 diff 교차 확인(프로세스) | ☐ |

## 교훈

- **같은 파일을 동시에 수정하는 병렬 PR은 "충돌 없이" 머지돼도 의도가 유실될 수 있다.** Git의 3-way 병합은 텍스트 충돌만 잡지, "한쪽이 추가한 가드를 다른 쪽 재작성이 누락"한 의미적 손실은 잡지 못한다. 같은 파일을 만지는 PR이 둘 이상이면 머지 후 핵심 가드가 살아남았는지 확인해야 한다.
- **프록시 뒤에서 `getRemoteAddr()`는 항상 내부 IP다.** 출발지 IP 기반 차단/레이트리밋은 내부 IP 화이트리스트를 가드로 전제해야 하며, 그 가드의 부재는 "일부 IP 오차단"이 아니라 "프록시 IP 차단 = 전체 트래픽 차단"으로 증폭된다.
- **임시 완화(Redis 키 삭제)는 증상만 지운다.** 404가 재누적되면 동일하게 재발하므로, 근본 원인(가드 부재·환경 미분리)을 함께 닫지 않으면 인시던트가 반복된다.
